# Создание персонального токена доступа (Personal Access Token) в Forgejo

## Что такое персональный токен?

Персональный токен — это безопасная альтернатива паролю для доступа к API, Git-репозиториям и другим ресурсам Forgejo. Токены можно ограничивать конкретными правами и репозиториями, что делает их идеальными для CI/CD, скриптов и автоматизации.

---

## Требования к версии

| Версия Forgejo | Доступные функции |
|----------------|-------------------|
| **v15.0 и новее** | ✅ Токены с ограничением по репозиториям<br>✅ Гибкие scope-права |
| **v10.0 – v14.x** | ⚠️ Только классические токены (без ограничения по репозиториям) |

## Способ 1: Создание через веб-интерфейс

### Шаг 1: Откройте настройки пользователя

1. Войдите в ваш Forgejo
2. Нажмите на **фото профиля** в правом верхнем углу
3. Выберите **Settings** (Настройки)

### Шаг 2: Перейдите в раздел токенов

В боковом меню выберите **Applications** (Приложения) → раздел **Manage Access Tokens**

### Шаг 3: Создайте новый токен

1. Нажмите кнопку **Generate New Token**
2. Введите **название токена** (например, `CI/CD pipeline token`)
3. Выберите **права доступа (scopes)**:

| Scope | Описание | Когда использовать |
|-------|----------|-------------------|
| `read:repository` | Чтение файлов, релизов, collaborators | Для клонирования и просмотра репозиториев |
| `write:repository` | Чтение и запись в репозиторий | Для push изменений, создания релизов |
| `read:packages` | Чтение пакетов из реестра | Для скачивания пакетов |
| `write:packages` | Публикация пакетов | Для загрузки пакетов |
| `read:issue` | Чтение задач (issues) | Для просмотра задач |
| `write:issue` | Создание и редактирование задач | Для автоматизации задач |

### Шаг 4: (Только Forgejo v15.0+) Выберите репозитории

Если вам нужен доступ **только к конкретным репозиториям**:
- Выберите опцию **"Limit to selected repositories"**
- Отметьте нужные репозитории из списка

### Шаг 5: Сохраните токен

1. Нажмите **Generate Token**
2. **Обязательно скопируйте** сгенерированный токен
3. Сохраните его в надёжном месте (например, в менеджере паролей)

> ⚠️ Токен показывается только один раз! При потере токена придётся создавать новый.

## Способ 2: Создание через API (для автоматизации)

```bash
curl -H "Content-Type: application/json" \
     -d '{"name":"my-api-token", "scopes":["read:repository"]}' \
     -u "ВАШ_ЛОГИН:ВАШ_ПАРОЛЬ" \
     https://ВАШ_ДОМЕН/api/v1/users/ВАШ_ЛОГИН/tokens
```

### Для двухфакторной аутентификации (2FA)

```bash
curl -H "Content-Type: application/json" \
     -H "X-Forgejo-OTP: 123456" \
     -d '{"name":"my-api-token", "scopes":["read:repository"]}' \
     -u "ВАШ_ЛОГИН:ВАШ_ПАРОЛЬ" \
     https://ВАШ_ДОМЕН/api/v1/users/ВАШ_ЛОГИН/tokens
```

### Ответ сервера

```json
{"id":1,"name":"my-api-token","sha1":"9fcb1158165773dd010fca5f0cf7174316c3e37d"}
```

## Использование токена

### Для доступа к API

```bash
# Через заголовок Authorization (рекомендуется)
curl -H "Authorization: token ваш_токен" \
     https://ВАШ_ДОМЕН/api/v1/repos/владелец/репозиторий

# Через параметр в URL (менее безопасно)
curl "https://ВАШ_ДОМЕН/api/v1/repos/владелец/репозиторий?token=ваш_токен"
```

### Для клонирования Git-репозитория

```bash
# Использование токена в URL
git clone https://ВАШ_ЛОГИН:ваш_токен@ВАШ_ДОМЕН/владелец/репозиторий.git

# Или через config
git config --global http.extraheader "Authorization: token ваш_токен"
```

### Для Docker-пакетов (если включено)

```bash
docker login https://ВАШ_ДОМЕН -u ВАШ_ЛОГИН -p ваш_токен
```

## Управление токенами

### Просмотр всех токенов

В интерфейсе: **Settings** → **Applications** → **Manage Access Tokens**

### Отзыв токена

**В веб-интерфейсе:** нажмите кнопку **Revoke** напротив нужного токена.

**Через API:**
```bash
DELETE /api/v1/users/{username}/tokens/{id}
```

## Безопасность

### ✅ Рекомендации

- Давайте токену минимально необходимые права
- Для CI/CD используйте ограничение по репозиториям (v15.0+)
- Храните токены в менеджере паролей
- Регулярно отзывайте неиспользуемые токены
- Используйте разные токены для разных сервисов

### ❌ Чего избегать

- Не встраивайте токены в код
- Не передавайте токены в логах
- Не используйте токены в публичных репозиториях
- Не используйте один токен для всего

|-------------|---------------|--------|
| Ограничение по репозиториям | ❌ | ✅ |
| Гибкие scopes (`read:repository`) | ❌ (только `repo`) | ✅ |
| Наследование прав администратора | ✅ | ❌ (безопаснее) |

## Устранение неполадок

| Проблема | Решение |
|----------|---------|
| `404 Not Found` | Токен не имеет доступа к этому репозиторию (v15.0+) |
| `403 Forbidden` | У токена недостаточно прав |
| `401 Unauthorized` | Неверный токен или формат заголовка |
| Токен не отображается после создания | Токен показывается только один раз, создайте новый |

## Примеры использования

### Для скрипта резервного копирования

```bash
#!/bin/bash
TOKEN="ghp_ваш_токен"
REPO="user/backup-repo"
API_URL="https://forgejo.example.com/api/v1/repos/${REPO}"

# Проверка доступа
curl -H "Authorization: token ${TOKEN}" ${API_URL}
```

### Для CI/CD (Jenkins, GitLab CI, Drone)

```yaml
# Пример для .drone.yml
steps:
  - name: clone
    image: alpine/git
    commands:
      - git clone https://your-token:${FORGEJO_TOKEN}@forgejo.com/user/repo.git
```
